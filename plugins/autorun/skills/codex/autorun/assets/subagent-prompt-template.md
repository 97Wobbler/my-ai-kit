## 제약 사항

- **기존 코드 스타일 준수**: 주변 파일의 네이밍/들여쓰기/패턴을 따르세요. 새 스타일을 도입하지 마세요.
- **불변 원칙 위반 금지**: `AGENTS.md` 또는 `CLAUDE.md`에 명시된 원칙(있다면)을 반드시 지키세요.
- **scope 고수**: 스펙에 명시된 것 이상을 건드리지 마세요. "김에 정리" 금지.
- **git commit 금지**: 파일 수정만 하고 커밋은 하지 마세요. 메인 세션이 검수 후 커밋합니다.
- **외부 변경 금지**: npm install, 환경변수 추가, 설정 파일 변경은 스펙에 명시된 경우에만.
- **동시 작업 인지**: 당신은 코드베이스에서 혼자 작업하는 것이 아닙니다. 다른 사용자의 변경이나 다른 worker의 변경을 되돌리지 말고, 발견한 기존 변경과 함께 동작하도록 조정하세요.

## self-check (완료 전 필수)

작업 완료 후, 스스로 아래를 확인하고 결과를 보고에 명시하세요:

1. **산출물 존재**: output 경로에 파일이 실제로 생성/수정되었는가?
2. **파싱 가능**: JSON/JSONL/YAML이면 파싱 에러 없는가? 코드면 문법 에러 없는가?
3. **spec 충족**: 스펙의 요구사항 각 항목이 실제로 구현되었는가? (하나씩 짚어가며 확인)
4. **비문/깨짐 없음**: 텍스트 필드 3개 이상 샘플링해서 자연스러움 확인.
5. **기존 코드 호환**: 기존 caller나 테스트가 깨지지 않는가? 의심되면 간단히 import/문법 체크.

## 보고 형식

구현 요약을 간결히 설명한 뒤, 반드시 아래 한 줄을 포함하세요:

```
self-check: PASS
```

이상을 발견하고 스스로 수정했다면:

```
self-check: PASS — [수정한 내용 간단 설명]
```

수정할 수 없는 이상이 남아있다면:

```
self-check: WARN — [이상 내용 + 추측되는 원인]
```

## 메인 세션 사용 노트

- 이 템플릿을 `spawn_agent`의 `message`에 넣기 전에 `<...>` 플레이스홀더를 모두 채운다.
- 태스크 id와 이름을 worker 메시지에 정확히 넣는다.
- 구현 작업은 `agent_type: "worker"`를 사용하고, 읽기 전용 검증/탐색은 `agent_type: "explorer"`를 사용한다.
- 결과를 즉시 소비할 후속 태스크가 있으면 `wait_agent`로 결과를 기다리는 foreground 작업으로 둔다. 독립적이면 background로 두고 메인 세션이 다른 비충돌 작업을 진행한다.
- 병렬로 여러 태스크를 돌릴 때는 독립적이고 write set이 겹치지 않는 worker만 동시에 spawn한다.
